# Repository Governance Defaults for Hermes Profile and Config Distributions

Use this reference when profile-builder work includes publishing or maintaining a Hermes configuration repository.

## Durable defaults

- Obtain the repository author's approved name and email before configuring Git. Represent them in public templates as `<repo-author-name>` and `<repo-author-email>`.
- Commit messages contain no `Co-authored-by` trailers, AI co-authoring trailers, or AI-attribution lines unless the repository owner explicitly requires them.
- New repository-authored skills default to `version: 0.1.0`, the approved repository author, and `license: Apache-2.0` unless the final specification overrides those fields.
- Public READMEs explain the generic profile/configuration system, update workflow, file layout, safety model, and future customization process.
- Public documentation omits sensitive profile contexts unless the user explicitly approves them for publication.

## Plan and final-spec checklist

Include:

1. A README task documenting future Hermes customization.
2. A repository-local Git identity task using operator-supplied environment values:
   ```bash
   git config user.name "$REPO_AUTHOR_NAME"
   git config user.email "$REPO_AUTHOR_EMAIL"
   ```
3. A validation task rejecting prohibited AI co-authoring trailers:
   ```bash
   git log --format=%B | grep -Ei 'co-authored-by|ai co-author|generated by ai' && exit 1 || true
   ```
4. A metadata task confirming every new skill has an approved author, semantic version, and license.

## Pitfalls

- Use `0.1.0`, not `v0.1.0`, when that default version is selected.
- Do not treat sensitive profile names as safe public README examples by default.
- Do not publish secrets, token stores, runtime logs, sessions, caches, databases, or profile-private memory.
