# Read-only plugin candidate classification

Use this procedure when classifying a plugin source inventory against hardened publication gates without invoking handlers or writing temporary homes.

## Evidence boundary

1. Pin repository `HEAD` and initial status before reading worktree files.
2. Hash the candidate tree canonically while excluding generated caches: sort files by relative POSIX path, then hash `relative_path + NUL + bytes + NUL` for every file. Recompute at the end.
3. Record existing `__pycache__`/`.pyc` separately; excluding them from the source hash does not make them pass source/install hygiene.
4. Recheck status during long audits. If a concurrent writer changes the publication worktree, stop treating its filesystem as committed evidence and inspect `HEAD` blobs with `git show`/`git ls-tree`. Do not revert or attribute concurrent changes to the audited commit.

## Per-candidate gates

Inspect every candidate for:

- provenance and source-profile gate;
- `plugin.yaml` identity, kind, declared tools, commands, hooks, CLI commands, skills, and env requirements;
- existence and quality of `tests/test_*.py`;
- TODO/FIXME, skipped/xfail tests, placeholder bodies, and mocks/stubs/fakes/spies/monkeypatches;
- Python limits: 200 physical lines per file, 30 per named construct, control-flow depth 3;
- bundled `SKILL.md` existence, complete frontmatter, minimum usable content, and manifest declaration parity;
- source/install hygiene, including generated bytecode;
- safe real behavior evidence.

A deterministic static failure makes the candidate `rejected` even when a separate real-behavior gate is blocked. Use `blocked` as the overall state only when no objective rejection exists and a required prerequisite cannot safely be exercised.

## Handler-free real manager parity

When registration itself is side-effect-free, use the installed real `PluginManager` without calling any registered handler, command, or hook:

1. Set `PYTHONDONTWRITEBYTECODE=1`.
2. Use the real manager's directory scanner on the candidate root.
3. Load each resulting real manifest through the manager's normal directory-module loader/registration path.
4. Compare actual registered tool, slash-command, hook, CLI-command, and qualified skill sets with `plugin.yaml`.
5. Confirm every skill registration resolves to an actual `SKILL.md` file.

Do not substitute a fake context. Do not mutate a real profile merely to enable candidates. Instead, copy the exact sanitized package into a disposable `HERMES_HOME`, enable only that candidate in the disposable config, set `PYTHONDONTWRITEBYTECODE=1`, and attribute registrations to that plugin. Compare **all declared surfaces separately**: tools, hooks, slash commands, CLI commands, and bundled skills. A tools-and-slash-commands-only parity helper is incomplete when `plugin.yaml` can declare hooks, CLI commands, or skills.

Before loading, statically inspect import and `register()` paths for network, subprocess, wallet/payment, credential, and filesystem side effects. Hash package bytes before and after registration. Also inventory the disposable home: real manager discovery may create benign bootstrap files such as `SOUL.md`; report those as disposable runtime writes rather than claiming the probe was globally write-free. If registration is not demonstrably inert, mark parity blocked.

### Safe representative handler probes

A deterministic static rejection remains `rejected` even if runtime behavior is otherwise healthy, but one representative real handler can still distinguish a loadable rejected package from dead code. Only do this after inert registration succeeds:

1. Use a fresh disposable home and exact sanitized package.
2. Remove inherited project-plugin enablement and set `PATH` to an empty disposable directory when the selected handler might discover or launch an external CLI.
3. Select a local/read-only operation such as `status`, `catalog`, `get`, `explain`, or `check`; never select payment, auth, send, mutation, refresh, install, or remote-update behavior merely to satisfy coverage.
4. Parse the documented JSON/text shape, require a clear missing-prerequisite result where applicable, and re-hash package bytes afterward.
5. Do not upgrade this positive probe into acceptance when tests, structure, declaration parity, or sanitized semantics already fail.

### Sanitizer overlap and semantic-corruption checks

Run the production sanitizer with the exact execution-time repository context and replacement arguments. Inspect more than secrets and absolute paths: replacement terms can overlap. For example, a private prefix may be a substring of an authorized source-profile name, so replacing the prefix first can turn a valid profile token into a malformed hybrid before the profile-specific replacement runs. Diff source versus projection per path and flag:

- hybrid placeholders embedded in commands, profile names, package IDs, or metadata;
- approved authorship rewritten to generic private placeholders outside explicitly preserved frontmatter;
- undeclared or corrupted CLI examples even when the sanitized package still imports;
- final-newline loss and generated caches as separate source-hygiene findings.

Hash both the canonical source tree and exact sanitized projection. If every candidate is objectively rejected, still report projection hashes and semantic defects; do not skip sanitizer inspection merely because publication is already blocked.

## Exact-object fallback

If worktree drift removes or rewrites packages during the audit:

- enumerate committed package paths with `git ls-tree -r --name-only HEAD`;
- read blobs with `git show HEAD:<path>`;
- run AST, line-count, placeholder, manifest, and bundled-skill checks in memory;
- report manager parity as unobserved for the exact object unless it was witnessed before drift or can be exercised without materializing files.

A committed package that passes provenance metadata but fails a newly hardened structural or real-test gate cannot be retained as `unchanged-last-known-good`. Last-known-good is relative to the current final gates, not merely to an older publication event.

## Reporting

Return two distinct matrices when both source candidates and already-public artifacts are in scope:

1. current source classification (`accepted`, `rejected`, `blocked`), including blocked sub-gates;
2. public last-known-good retention (`may remain` / `may not remain`) against the same final gates.

Use shared evidence codes only after defining exact `file:line` patterns, and still list every named candidate as its own row. Report commands actually run, handlers explicitly not invoked, initial/final hashes, HEAD, status drift, and pre-existing cache counts.
